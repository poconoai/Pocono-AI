"Your patient and client data never leaves the building."

Mechanism
The Sentinel Node runs Ollama locally on customer-owned hardware, bound to 127.0.0.1:11434. The inference model never makes outbound calls. There is no API key to a vendor service. Removing the network cable does not degrade functionality — we test for this.
Evidence
# Ollama config — bound to localhost only
SENTINEL_OLLAMA_URL = "http://localhost:11434"
# If this host had outbound connectivity, the deployment
# would still work. If localhost binding fails, it won't.
Citation
Ollama (open source) Trust & Security →

How to falsify this claim

Run tcpdump -i any 'host not 127.0.0.1' on the deployed Sentinel Node during a live query. If you observe outbound packets carrying inference data, our claim is wrong. Customers may run this themselves; results are reproducible.

"Audit log entries cannot be modified or deleted — not even by us."

Mechanism
On hourly rotation, closed log segments receive the Linux chattr +i immutable bit. A SHA-256 hash of each segment is appended to a hash-chain log, which itself becomes immutable on rotation. Modifying any prior segment breaks the chain, and the modification itself appears in the system audit trail.
Evidence
# Excerpt from logrotate post-rotate hook
sha256sum /var/log/sentinel/audit.log.1 \
    >> /var/log/sentinel/hash-chain.log
chattr +i /var/log/sentinel/audit.log.1
chattr +i /var/log/sentinel/hash-chain.log
Citation
HIPAA §164.312(b) NIST SP 800-53 AU-9 README.md (public)

How to falsify this claim

Try to write to a closed segment. The kernel will deny the operation regardless of user privilege, including root, until chattr -i is explicitly run — which itself appears in auditd as a forensic event.

"Every query and response is cryptographically signed before the user sees it."

Mechanism
The audit wrapper sits between the user and the LLM. Before the model is invoked, a QUERY_IN event is HMAC-SHA256-signed with a 256-bit key and fsync()'d to disk. After inference but before the user sees the response, a RESPONSE_OUT event is signed and persisted. Both events share an hmac_query field that links them.
Evidence
def log_query(user_id, prompt, model):
    hmac_query = hmac_sign(prompt, SENTINEL_HMAC_KEY)
    event = {
        "type": "QUERY_IN",
        "ts": utc_now(),
        "hmac_query": hmac_query,
        "user_id": user_id,
        "model": model,
    }
    write_and_fsync(event)  # before LLM invocation
Citation
HIPAA §164.312(a)(2)(i) HIPAA §164.312(b) RFC 2104 (HMAC)

How to falsify this claim

Tamper with any logged event field. Re-running sentinel_audit_wrapper.py --verify against that segment will exit non-zero and print the offending record.

"Air-gapped by topology, not by policy."

Mechanism
The deployment specification places the Sentinel Node on a VLAN with no default gateway to the internet. Inference services bind to localhost. Even if a malicious actor exfiltrated the HMAC key, there is no route over which to ship the data. You cannot lose what cannot leave.
Evidence
Validated at deployment via ip route showing no default route, and ss -tlnp confirming no externally-bound listeners. Customers receive a network topology diagram as part of the SLA.
Citation
HIPAA §164.312(e) — Transmission Security Architecture →

How to falsify this claim

From the Sentinel Node, attempt curl https://1.1.1.1. The connection should fail. If it succeeds, the deployment is misconfigured; we treat that as a Severity-1 incident.

"We don't phone home. Ever."

Mechanism
No analytics SDK, no telemetry library, no auto-update beacon, no usage reporter. The Sentinel Node software has zero outbound dependencies after install. Updates are delivered as signed packages a customer chooses to apply — never pulled.
Evidence
Ship manifest is published. requirements.txt for the audit wrapper contains requests and python-dotenv; requests is used solely to call the localhost Ollama endpoint. No analytics packages, no Sentry, no Datadog agent, no anything.
Citation
Zero Telemetry →

How to falsify this claim

Run packet capture for 24 hours on the Sentinel Node. Any outbound connection attempt other than to localhost falsifies this claim. We've never observed one. We invite the test.

"HIPAA technical safeguards: not aspirational, mapped."

Mechanism
Each technical safeguard in 45 CFR §164.312 is implemented by a specific architectural element, not by a policy document. We publish the mapping and the corresponding code or configuration line.
Evidence
(a)(1) Access Control → Linux user/group + SSH key auth, no shared accounts
(a)(2)(i) Unique User ID → user_id field on every audit record
(b) Audit Controls → HMAC-signed audit wrapper, see Receipt 03
(c)(1) Integrity → Hash-chain + immutable bit, see Receipt 02
(d) Person/Entity Auth → SSH keys, no password fallback
(e)(1) Transmission Security → No transmission, see Receipt 04
Citation
45 CFR §164.312 Generate Attestation →

"You own the hardware. We don't have a key."

Mechanism
Title to the Sentinel Node hardware transfers to the customer at deployment. There is no remote management channel, no vendor backdoor, no SSH key held by Pocono AI. The HMAC key is generated on the customer's machine and held only by the customer.
Evidence
sentinel_audit_wrapper.py --generate-key creates a 256-bit key locally, prints it once, and never transmits it. The .env file is in .gitignore for a reason: it is operational secret material the vendor has no business possessing.
Citation
SLA → Terms of Service →

How to falsify this claim

Audit the Sentinel Node for any user account, SSH key, or service that traces back to Pocono AI infrastructure. None will be present. If one is, it is a deployment defect, not policy.

"Your data is never used to train any model."

Mechanism
The model weights deployed to your Sentinel Node are static. Inference does not modify weights. There is no telemetry channel back to Pocono AI or to the upstream model authors. Fine-tuning, if a customer requests it, runs locally on customer hardware against customer data — the resulting weights remain on customer hardware.
Evidence
Ollama runs models in pure inference mode. No train() code path is invoked at runtime. The model files are read-only at the OS level after install.
Citation
Terms § Data Use →

"Your audit history survives a full hardware replacement."

Mechanism
Closed audit segments and the hash chain can be backed up to an air-gapped storage medium of the customer's choice. On hardware replacement, the new Sentinel Node verifies the chain end-to-end before resuming operations. The HMAC key required for verification is the customer's, not ours.
Evidence
sentinel_audit_wrapper.py --verify <segment> exits with status 0 on a clean segment and 1 on any integrity issue. The verification is a single command; it is not a managed service we run for you.
Citation
README →

"Even a compromised admin account cannot rewrite history."

Mechanism
A root-level attacker would need to: (1) run chattr -i on the target log segment, (2) modify the segment without breaking the per-line HMAC, (3) regenerate the SHA-256 entry in the hash-chain log, (4) recompute every subsequent hash-chain entry, and (5) hide the chattr call from auditd. Each step is observable. The chain breaks at the first inconsistency.
Evidence
The HMAC key is required to forge new entries that pass verification. The HMAC key never appears in any log entry — it lives only in the customer's .env or secrets manager. Compromising the running system does not yield the key unless the attacker also extracts memory at exactly the right moment during a request.
Citation
NIST SP 800-53 AU-9 CIA Triad — Integrity

How to falsify this claim

Modify a closed segment without breaking --verify. We have not seen this done. We do not believe it is feasible without the customer's HMAC key, which we do not hold.
