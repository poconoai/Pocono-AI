Physical & Internal Security Architecture

The Perimeter Is Secure
From the Inside Out

The Sentinel Node defeats cloud exfiltration by architecture. But a compliance officer's next question is the right one: what stops an insider threat? Here is the complete answer — physical port control, encrypted storage, and role-based access control.

The Complete Threat Model

Zero cloud exfiltration eliminates the external threat vector. But a rigorous security posture must also address the internal one: an employee with physical access to the hardware. The Sentinel Node's security architecture layers three defenses against insider threats — port lockdown, at-rest encryption, and role-based access controls.

🔒

Physical Port Lockdown

Unauthorized USB devices are hardware-blocked. Signed media only. No walk-out-with-a-USB attacks.

💾

Encrypted At-Rest Storage

Full-disk encryption renders the vectorized database unreadable if the hardware is physically removed.

👤

Role-Based Access Control

Query access is scoped by role. A paralegal cannot reach clinical records. A billing clerk cannot reach privileged case files.

■ Layer 1 · Physical Port Security

An Angry Paralegal With a USB Drive Is Not a Threat Vector

The Sentinel Node's USB policy enforces the same cryptographic trust model used for model weight updates — applied to every port on the machine. This is not a software policy that can be bypassed by an administrator. It is enforced at the kernel level.

⊘ Blocked — Unauthorized USB
  • Unsigned USB media — system refuses to mount
  • Unknown device IDs — blocked at kernel level via udev rules
  • Mass storage devices not on the approved hardware whitelist
  • Attempted bypass via USB-C or Thunderbolt adapters
  • Every rejected insertion attempt is logged to the audit trail
✓ Permitted — Authorized Media Only
  • Cryptographically signed update packages (Ed25519)
  • Whitelisted hardware device IDs — pre-approved during deployment
  • Package signature verified before any filesystem mount
  • SHA-256 hash checked against Secure Portal value
  • Successful and failed attempts both logged immutably

What this means in practice: An employee who plugs an unauthorized USB drive into the Sentinel Node will find that nothing happens — the device is not recognized, not mounted, and not accessible. The attempt is silently logged with timestamp and device identifier. The next time the IT administrator reviews the audit trail, the event is visible. The data goes nowhere.

■ Layer 2 · Encrypted At-Rest Storage

If Someone Walks Out With the Hardware, They Get an Encrypted Brick

Full-disk encryption (LUKS/dm-crypt on the OS layer, with hardware-accelerated AES-256) is applied to every storage volume on the Sentinel Node — the OS partition, the Qdrant vector database, the RAG document index, and the audit log. An attacker who physically removes the drives gets encrypted data that is computationally infeasible to recover without the key.

AES-256
Full-disk encryption standard. Hardware-accelerated on Dell Precision processors.
LUKS2
Linux Unified Key Setup — the industry standard for enterprise disk encryption on Linux.
Key Custody
Encryption keys managed by the client organization — not held by Pocono AI. Pocono AI cannot decrypt your data.

What gets encrypted: The Qdrant vector database (your indexed patient records / case files), the raw document store, the 90-day immutable audit log, all OS configuration and credentials, and all temporary inference artifacts. The encryption key is held by the client organization and is required for every system boot — if the hardware is powered off and moved, it will not boot without the key.

■ Layer 3 · Role-Based Access Control (RBAC)

The Right Data to the Right Role — Nothing More

Even with full access to the local network, a staff member can only query data appropriate to their role. RBAC is configured during the 60-Day Proof of Value deployment and scoped to the practice's specific staff structure. Query permissions are enforced at the API layer — not just in the UI.

Role Can Query Cannot Query Audit Logged
Physician / NP (Tier A) Full patient record index, clinical summaries, referral history Billing admin records, system config Every query logged
Clinical Support (Tier B) Medication history, prior auth support documents, triage notes Diagnosis summaries requiring licensed review, full clinical notes Every query logged
Administrative (Tier C) Billing context, insurance policy docs, scheduling information All clinical records, medication histories, diagnosis documents Every query logged
IT Administrator System configuration, audit log review, update application Patient records, clinical data (IT access is scoped to infra only) All admin actions logged

How RBAC is enforced: Each authenticated workstation session carries a role token issued during deployment configuration. The Sentinel Node's API layer validates this token on every query — not just at login. A billing clerk cannot construct a query that returns clinical records, regardless of how they phrase it. The restriction is at the data layer, not the interface layer. Role tokens are rotated during the quarterly SLA review and can be revoked immediately by the IT administrator without rebooting the system.

The 90-Day Immutable Audit Trail

Every event — query, USB insertion attempt, role authentication, update application, and system boot — is written to a tamper-resistant local log. The log is write-once during its 90-day retention window: entries cannot be modified or deleted, even by the IT administrator. After 90 days, entries are archived per the client's data retention policy.

90 Days
Immutable retention window for all system events
Write-Once
No admin can modify or delete audit entries during the retention window
Local Only
Audit log stored on the Sentinel Node — never transmitted to Pocono AI or any external system
OCR Ready
Log format meets HIPAA technical safeguard audit requirements for OCR review
